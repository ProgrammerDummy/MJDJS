WEEK 5: Scheduler Server — State, Dispatch Loop, gRPC Handlers
================================================================

MILESTONE
---------
Scheduler server accepts job submissions, queues by priority,
dispatches to registered workers, tracks job state —
all tested with real gRPC integration tests.

HOURS: 6/day | SPLIT: 1h design · 1h reading · 4h coding + testing

--- READ ---

1. DDIA Chapter 8 — Section 8.3 (Unreliable Clocks)
   - Read before storing any timestamps
   - Monotonic clocks for durations, wall clocks for timestamps
   - Never compare timestamps from two different machines for ordering
   - Use logical ordering instead of timestamp ordering

2. tokio mini-redis source code
   - URL: github.com/tokio-rs/mini-redis
   - Read ALL source files before writing your server
   - src/lib.rs, src/server.rs, src/connection.rs, src/db.rs
   - Every design decision has a reason — understand the reasons
   - You are NOT copying it. You are internalizing the patterns.

--- DESIGN ---

3. Draw SchedulerState on paper BEFORE writing any code
   Fields to design:
   - job_queue: what type? (BinaryHeap needs exclusive write access)
   - running_jobs: what type? (hot read path — concurrent reads)
   - completed_jobs: what type? (same as running_jobs)
   - workers: what type? (concurrent reads and writes)
   - metrics counters: what type? (lockless atomic increments)

   Answers:
   - job_queue: Arc<Mutex<BinaryHeap<QueuedJob>>>
   - running_jobs: Arc<DashMap<JobId, RunningJob>>
   - completed_jobs: Arc<DashMap<JobId, CompletedJob>>
   - workers: Arc<DashMap<WorkerId, WorkerInfo>>
   - counters: Arc<AtomicU64> for submitted/completed/failed

   Write a comment on EVERY field explaining why you chose that type.
   A wrong choice here requires a full refactor after week 8 benchmarks.

--- BUILD ---

4. Implement SchedulerState with correct concurrent data structures
   - DashMap for maps read concurrently (hot read path)
   - Mutex<BinaryHeap> for queue (exclusive during dequeue)
   - Keep lock duration MINIMAL — never hold while doing I/O
   - Arc<AtomicU64> counters: total_submitted, completed, failed

5. Implement the dispatch loop as a dedicated tokio task
   - Loop forever:
       1. Lock queue
       2. Peek at highest-priority job
       3. Find available worker that supports job_type
       4. If found: dequeue, unlock, dispatch to worker
       5. If not found: unlock, sleep 10ms, retry
   - CRITICAL: never hold lock while doing I/O
   - If selected worker disconnects before dispatch:
     return job to queue
   - Log every dispatch: job_id, job_type, priority,
     worker_id, queue_depth, dispatch_latency_us

6. Implement SubmitJob, GetJobStatus, CancelJob gRPC handlers
   - SubmitJob: validate → assign UUID v7 → push to queue → return ID
   - GetJobStatus: check running_jobs, then completed_jobs, return state
   - CancelJob:
       If Queued: remove from queue
       If Running: send cancel signal to worker via oneshot
   - Test: submit 100 jobs, cancel 30 while Queued
     → verify exactly 70 execute
   - Test: CancelJob on already-completed job → graceful error, not panic

7. Set up CI and commit discipline
   - GitHub Actions: cargo test + clippy + fmt on every push
   - Rule: main branch ALWAYS passes cargo test
   - A failing CI build is a blocker — fix before sleeping
   - Add cargo-deny check to CI

--- TEST ---

8. Integration tests (3 mandatory tests)
   - Test 1: Priority ordering
     Submit 1 high-priority job AFTER 10 low-priority
     → high-priority must execute first
   - Test 2: Concurrent submissions
     100 concurrent clients × 10 jobs = 1000 total
     → server must not panic or deadlock
   - Test 3: Cancellation
     Cancel 50 queued jobs → verify none execute
   - Add a 10-second randomized stress test — catches timing bugs
   - All integration tests must run under 5 seconds total

CRATES THIS WEEK
----------------
tokio
tonic
dashmap
uuid (v7 feature)
tracing
tracing-subscriber

RISKS
-----
[CRITICAL] Holding queue lock while sending to worker channel —
if channel is full (backpressure), you deadlock.
Always: lock → dequeue → unlock → THEN send.

[WARNING] Using Mutex<HashMap> instead of DashMap for running_jobs —
GetJobStatus is the hot read path. A global lock serializes all reads.
You will see this in week 8 benchmarks. Use DashMap.

WEEK 6: Worker Agent — Registration, Heartbeat, Failure Detection
================================================================

MILESTONE
---------
Worker agents that register, receive jobs, execute them, send
heartbeats — and the scheduler correctly detects SIGKILL and
reassigns in-flight jobs within 20 seconds. Proven by SIGKILL
chaos test passing 10 consecutive times.

HOURS: 6/day | SPLIT: 1h reading · 1h design · 4h coding + testing

--- READ ---

1. AWS Builders' Library — Heartbeat and Constant Work
   - URL: aws.amazon.com/builders-library/reliability-and-constant-work
   - Key: a heartbeat that only fires when there's work is NOT
     a heartbeat — it's a work notification
   - Key: heartbeat must fire at constant rate regardless of load
   - How does constant work prevent head-of-line blocking?

2. Phi Accrual Failure Detector — Introduction and Sections 1–3
   - Google: "phi accrual failure detector"
   - Read intro + sections 1–3 ONLY (skip the math)
   - Why is a fixed threshold inferior to an adaptive one?
   - Cassandra and Akka use this — you're implementing fixed threshold
   - Know its limitations so you can explain them in interviews

--- DESIGN ---

3. Design the heartbeat + lease mechanism on paper
   - Worker sends heartbeat every 5 seconds
   - Scheduler records last_heartbeat_at per worker
   - Background task scans every 10 seconds:
       if now - last_heartbeat_at > 15 seconds
       → mark worker Dead
       → reclaim all running_jobs from that worker
       → requeue them as Queued
   - 15 seconds = 3 missed heartbeats (minimum safe threshold)
   - Threshold must be > 2× interval to tolerate 1 missed packet
   - 3× (15s/5s) is safe

   Draw the sequence diagram:
   Worker → Heartbeat RPC → Scheduler (update last_heartbeat_at)
   Background task → scan → if stale → Dead → reclaim → requeue

--- BUILD ---

4. Implement worker registration with capability declaration
   - RegisterWorker RPC sends:
       worker_id (deterministic: hash of hostname + PID)
       hostname
       job_types_supported: Vec<String>
       max_concurrent_jobs: u32
       tags: map<string,string>
   - Deterministic worker_id: same worker reconnecting = same ID
   - job_types_supported IS your routing key
   - Test: 5 workers with different capabilities
     → verify scheduler only routes matching job_types

5. Implement bidirectional streaming heartbeat RPC
   - ONE persistent bidirectional stream per worker (not repeated unary)
   - Worker sends Heartbeat { worker_id, current_jobs, capacity }
     every 5 seconds
   - Scheduler receives → updates last_heartbeat_at in workers DashMap
   - Handle stream reconnect: if stream drops, worker reopens + re-registers
   - Test: disconnect stream for 16 seconds
     → scheduler marks worker Dead

6. Implement failure detection background task
   - tokio::spawn a task that runs every 10 seconds
   - Scan all workers
   - For each where now - last_heartbeat_at > 15s:
       mark state=Dead
       move all running_jobs → Queued state
       remove from active pool
   - MUST NOT hold workers lock while modifying running_jobs
     (potential deadlock)
   - Log: worker_id, last_heartbeat_at, jobs_reclaimed, requeue_count

7. Implement job execution with cancellation in worker
   - Worker receives job via RequestWork RPC
   - Spawns tokio task to execute it
   - Passes CancellationToken to execution task
   - On cancel signal from scheduler: cancel token, exit cleanly
   - For testing: SleepJob executor (sleeps for payload_duration_ms)
   - Report result via ReportResult RPC
   - Test: scheduler sends cancel during sleep
     → worker exits cleanly within 1 second

--- TEST ---

8. SIGKILL chaos test — 10 consecutive runs, zero job loss
   Setup: scheduler + 3 workers, submit 30 jobs (10 per worker)
   Action: send SIGKILL to worker-2
   Verify: all 30 jobs eventually complete
   Record:
   - Time from SIGKILL to scheduler detecting death (target < 20s)
   - Time to full reassignment (target < 25s)
   Run: 10 consecutive times
   Rule: ANY run where a job is lost = bug. Stop and fix before continuing.

CRATES THIS WEEK
----------------
tokio
tonic
tokio-util (CancellationToken)
tracing

RISKS
-----
[CRITICAL] Using unary Heartbeat RPCs instead of bidirectional
streaming — 1 RPC per heartbeat at 5s interval = 12 RPCs/min.
At 100 workers = 1200 RPCs/min just for heartbeats.
Use ONE persistent streaming RPC per worker.

[CRITICAL] Testing with SIGTERM instead of SIGKILL —
graceful shutdown (SIGTERM) is easy. SIGKILL is how workers
actually die in production. Chaos test must use SIGKILL ONLY.

[WARNING] Job execution leaking resources on cancellation —
if executor spawns child processes or opens file handles,
cancellation must clean them up. Write a resource leak test.

NOTES
-----
- A slow worker and a dead worker are indistinguishable.
  Your threshold (15s) is a bet: genuine death = wait up to 15s.
  Slow worker = may falsely declare dead and re-run its jobs.
  This is the at-least-once tradeoff. Know it, test it, explain it.

WEEK 7: Retry Logic, Dead-Letter Queue, SQLite Persistence
================================================================

MILESTONE
---------
Failed jobs retry with correct exponential backoff + jitter,
exhaust retries and dead-letter correctly. Scheduler recovers
all incomplete jobs after a SIGKILL restart — proven by 10
consecutive chaos test runs.

HOURS: 7/day | SPLIT: 1h reading · 1.5h design · 4.5h coding + chaos

--- READ ---

1. "You Cannot Have Exactly-Once Delivery" — Brave New Geek
   - Google that EXACT title
   - Read in full — most important conceptual reading of project
   - Why is exactly-once impossible across scheduler → worker boundary?
   - What does "idempotent" mean?
   - Why must your job executors be idempotent?
   - Save this post — you will reference it in every interview

2. Stripe Engineering Blog — Idempotency Keys
   - URL: stripe.com/blog/idempotency
   - Read fully
   - How Stripe makes at-least-once safe
   - Your job executors need the same pattern
   - Document in README: "This scheduler provides at-least-once
     delivery. Job executors must be idempotent."

--- DESIGN ---

3. Design retry flow and DLQ on paper
   On job failure:
   - Check retry_count vs max_attempts
   - If retries remain:
       compute next_retry_at = now + RetryPolicy::next_delay(retry_count)
       set state = Retrying
       increment retry_count
       add to retry queue (sorted by next_retry_at, NOT main priority queue)
   - If no retries remain:
       set state = DeadLettered
       move to DLQ

   Retry queue: separate structure sorted by next_retry_at
   Background task: moves Retrying jobs back to Queued when now >= next_retry_at

--- BUILD ---

4. Implement retry logic using your week-2 RetryPolicy
   - On ReportResult with status=Failed:
       look up job's RetryPolicy
       call next_delay(retry_count)
       if Some(delay): set Retrying, next_retry_at = now+delay, increment count
       if None: set DeadLettered
   - Test: max_attempts=3, executor always fails
     → retries exactly 3 times → dead-letters
   - Test: delays are exponentially increasing with jitter (log them)
   - Test: job succeeds on retry-2 → Succeeded, not DeadLettered

5. Implement dead-letter queue with inspection and requeue RPCs
   - DLQ: DashMap<JobId, DeadLetteredJob>
   - DeadLetteredJob contains:
       original job, failure reason, all attempt logs, final_failed_at
   - Add gRPC RPCs: ListDeadLettered, RequeueFromDLQ
   - RequeueFromDLQ: reset retry_count=0, re-enqueue at original priority
   - DLQ size limit: when full, drop oldest entry, log warning
   - Test: fill DLQ to capacity → verify oldest dropped
   - Test: RequeueFromDLQ → verify job executes from retry_count=0

6. Implement SQLite persistence with sqlx in WAL mode
   - Every state transition writes to SQLite:
       INSERT on submit
       UPDATE on each state change
   - Enable WAL mode: PRAGMA journal_mode=WAL
     (allows concurrent reads during writes — NON-NEGOTIABLE)
   - Schema: jobs table with all Job fields
     + state_history as JSON
     + index on (state, priority, created_at)
   - On restart: SELECT all jobs WHERE state IN
     ('Queued', 'Running', 'Retrying') → recover these
   - Running jobs on restart → Queued
     (we don't know if worker finished — at-least-once)
   - Test: write 10,000 jobs, SIGKILL, restart
     → verify all 10,000 recovered
   - Measure: recovery time for 10,000 jobs (target < 2 seconds)

--- TEST ---

7. Full restart chaos test — 10 consecutive runs
   Submit 500 jobs
   SIGKILL scheduler after 200 submitted and dispatching
   Restart scheduler
   Verify: all 500 eventually complete (200 from before kill, 300 from recovery)
   Run: 10 consecutive times
   Rule: ANY lost job = bug. Stop and fix.
   Record: restart time (target < 3s), recovery time (target < 5s)

8. Retry + persistence combined chaos test
   Submit 100 jobs where 50% fail on first attempt
   SIGKILL scheduler mid-run
   Restart
   Verify:
   - All jobs complete or dead-letter correctly
   - retry_count preserved across restart (stored in SQLite)
   - retry_count in SQLite = retry_count in memory for every job

CRATES THIS WEEK
----------------
sqlx (sqlite, runtime-tokio-rustls features)
tokio
tracing

RISKS
-----
[CRITICAL] At-least-once means a job can run TWICE if worker
completes it but scheduler crashes before recording completion.
Document this explicitly. Make all example executors idempotent.

[CRITICAL] SQLite without WAL mode — default journal mode blocks
concurrent reads during writes. GetJobStatus (hot path) blocks
during every state transition. Enable WAL mode before any benchmarks.

NOTES
-----
- This post explains the fundamental constraint you're working within.
  Being able to say "my scheduler provides at-least-once delivery,
  which is the practical guarantee for distributed systems, and
  requires idempotent executors" is a precise, sophisticated answer.
