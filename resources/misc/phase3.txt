--- BUILD ---

4. Implement leader election using etcd leases
   On startup:
   - Grant a lease with 10-second TTL
   - Try to PUT scheduler/leader with the lease ID
   - If key already exists → become STANDBY:
       watch the key for changes
       when key disappears → try to become leader
   - If PUT succeeds → become LEADER:
       start dispatch loop
       start heartbeat checker
       keep lease alive (renew every 3 seconds)
   - On losing leadership (lease expired):
       stop dispatch loop
       stop accepting new submissions
       wait for new leader

   Use etcd-client crate: grant_lease, keep_alive, put with lease, watch

WEEK 10: Worker Routing, Priority Aging, Resource-Aware Dispatch
================================================================

MILESTONE
---------
Tag-based routing, resource-aware scheduling, and priority
starvation prevention all working and tested on school hardware
with heterogeneous workers.

HOURS: 2 hrs/day weekdays · 5 hrs/day weekend
SPLIT: 50% implementation · 50% testing

--- READ ---

1. Kubernetes Scheduler — Filtering and Scoring
   - URL: kubernetes.io/docs/concepts/scheduling-eviction/kube-scheduler
   - Read: "Filtering" and "Scoring" sections specifically
   - Kubernetes scheduling = filter phase then score phase
   - Your tag matching (job requirements ⊆ worker tags)
     IS Kubernetes "node selector"
   - Your resource checking IS Kubernetes resource requests
   - After reading: what is "taints and tolerations"?
     Could you add this concept?

--- BUILD ---

2. Implement tag-based worker routing
   - Workers declare tags: {'gpu':'true', 'region':'us-east', 'memory_gb':'32'}
   - Jobs declare requirements: {'gpu':'true'}
   - Dispatch ONLY routes job to worker where:
     ALL of job.requirements keys present in worker.tags with matching values
   - If no matching worker available: job waits in queue
   - Test: 3 GPU workers, 3 CPU workers, 10 GPU jobs, 10 CPU jobs
     → GPU jobs must ONLY run on GPU workers
   - On school servers: label machines based on their actual hardware

3. Implement resource-aware scheduling
   - Workers declare: cpu_cores: u32, memory_mb: u32
   - Jobs declare: requires_cpu: u32, requires_memory_mb: u32
   - Track allocated_cpu and allocated_memory per worker
   - Dispatch ONLY if worker has enough free capacity
   - Release resources on job completion
   - Test: worker has 4 cores, submit 6 jobs × 1 core each
     → max 4 concurrent, 5th and 6th wait
   - Test: worker has 8GB RAM, job requires 10GB
     → must NEVER dispatch to that worker

4. Implement priority starvation prevention with aging
   - A low-priority job queued > aging_threshold_seconds
     gets effective_priority boosted by 1 every aging_interval_seconds
   - Background task runs every 30 seconds
   - Scans queued jobs, applies boosts
   - effective_priority: separate field from original priority (immutable)
   - BinaryHeap sorts by effective_priority
   - When boosted: remove and re-insert job (BinaryHeap doesn't support mutation)
   - Test: submit 1000 high-priority jobs + 1 low-priority job
     → low-priority job must eventually run

--- TEST ---

5. Heterogeneous worker routing test on school hardware
   Setup: 2 workers tagged gpu=true, 3 workers without
   Submit: 20 GPU jobs + 30 CPU jobs
   Verify: GPU jobs ONLY run on GPU workers
   Check: scheduler logs show matched tags for every dispatch
   Check: SQLite jobs table — every GPU job has worker_id
          of a GPU-tagged worker

WEEK 11: Rate Limiting and Fan-Out Jobs
================================================================

MILESTONE
---------
Per-job-type rate limiting and global token bucket enforced under
load. Fan-out jobs with parent completion semantics working and tested.

HOURS: 2 hrs/day weekdays · 5 hrs/day weekend
SPLIT: 60% implementation · 40% testing

--- READ ---

1. Stripe Engineering Blog — Designing Robust Rate Limiting
   - Google: "stripe rate limiting blog"
   - Read fully
   - Token bucket vs sliding window — which is better for bursty traffic?
   - Why jitter matters in rate limiters too
   - How would you extend your single-node rate limiter to work
     across 2 scheduler nodes? (Good interview question — have an answer)

2. Google SRE Book — Chapter 21: Handling Overload
   - URL: sre.google/sre-book/handling-overload
   - "Utilization signal" — use queue depth as a rate limiting signal
   - What is load shedding?
   - When should your scheduler REJECT new submissions instead of queuing?

--- BUILD ---

3. Implement per-job-type rate limiting
   - Config: max_concurrent_per_type: HashMap<String, u32>
   - In dispatch loop, before dispatching job of type X:
     count running jobs of type X
     if count >= limit: skip this job, try next in queue
   - Store running_count_by_type: DashMap<String, AtomicU32>
     increment on dispatch, decrement on completion
   - Test: limit resize_image to 3 concurrent, submit 10
     → verify max 3 run at once at any time
   - Expose rate_limit_utilization{job_type=X} gauge in metrics

4. Implement global token bucket rate limiter
   - TokenBucket { capacity: u32, tokens: AtomicU32,
                    last_refill: AtomicU64, refill_rate_per_sec: u32 }
   - Before each dispatch: try to consume 1 token
   - If empty: wait until next refill
   - Refill continuously (NOT in large batches — causes bursting)
   - Use INTEGER tokens (not float — float drift over long runtimes)
   - Test: limit=10 jobs/sec, submit 100 jobs, 20 workers
     → measure actual throughput = 10 ± 0.5 jobs/sec
   - Test burst: send 1000 jobs at once
     → verify they drain at refill_rate, not all at once

5. Implement fan-out jobs (parent-child, depth=1 MAXIMUM)
   - Running job submits child jobs via ScheduleChildren RPC
   - Parent remains Running until ALL children complete
   - If any child fails + exhausts retries: parent = Failed
   - Store parent_id: Option<JobId> on each Job
   - Depth limit: children CANNOT spawn grandchildren (depth=1)
     Detect and reject via ancestry chain check
   - Test: parent spawns 5 children, 2 fail all retries
     → parent = Failed
   - Test: parent spawns 5 children, all succeed
     → parent = Succeeded
   - Document in README: "Fan-out supports depth=1 (parent-child).
     Full DAG dependencies are not supported."

--- TEST ---

6. Rate limiter accuracy test under sustained load
   Submit 600 jobs over 60 seconds, global limit = 10/sec
   Measure actual throughput every 10 seconds
   Target: 10 ± 1 jobs/sec in EVERY 10-second window
   Plot throughput over time — should be FLAT, not spiky
   Spiky = token bucket refilling in batches (fix: refill continuously)
   Below 9/sec = rate limiter too conservative (fix: token consumption logic)

CRATES THIS WEEK
----------------
tokio
dashmap
governor (read docs — decide: use it or roll your own token bucket)

RISKS
-----
[WARNING] Fan-out circular dependency — job A spawns job B which
spawns job A. Detect cycles: maintain Set<JobId> of ancestor IDs
and reject any spawn that would create a cycle.
Unchecked cycles = infinite recursion in completion listener.

[WARNING] Token bucket refill creating thundering herd — if you
refill ALL tokens at once every second, you get a burst every second.
Refill continuously: on each dispatch attempt, compute tokens since
last_refill, add them, consume one.

NOTES
-----
- Fan-out scope is deliberately limited to depth=1. Full DAG support
  (arbitrary job dependencies) is a 4-week project by itself.
  Document the limitation honestly. Explain in interviews why
  depth=1 is the right starting tradeoff.

WEEK 12: Cron Scheduling and Job Templates
================================================================

MILESTONE
---------
Cron jobs fire at correct times, survive scheduler restarts,
and do NOT double-fire after leader failover — proven by
controlled failover-during-fire-window test (5 runs).

HOURS: 2 hrs/day weekdays · 5 hrs/day weekend
SPLIT: 40% reading + design · 60% implementation + testing

--- READ ---

1. "Falsehoods Programmers Believe About Time"
   - Google that EXACT title
   - Read BEFORE touching any cron or timestamp code
   - NON-NEGOTIABLE
   - Key falsehood: "a minute always has 60 seconds" (DST)
   - Key falsehood: "two machines agree on the current time"
   - RULE: store ALL timestamps as UTC in SQLite
     Convert to user timezone ONLY at display time. NEVER at storage.

2. DDIA Chapter 8 — Monotonic vs Wall Clocks section
   - Book: Designing Data-Intensive Applications (Kleppmann)
   - Monotonic clocks for durations (heartbeat intervals, retry delays)
   - Wall clocks for human-readable timestamps ONLY
   - cron tick loop: use wall clock (SystemTime) to check next_fire_at
   - heartbeat checker: use std::time::Instant (monotonic)
   - NEVER use Instant in SQLite — no absolute meaning across restarts

3. Kubernetes CronJob — Concurrency Policy
   - URL: kubernetes.io/docs/concepts/workloads/controllers/cron-jobs
   - Read: "Concurrency Policy" section specifically
   - Three policies: Allow, Forbid, Replace
   - Choose and implement one as your default
   - How does Kubernetes handle double-fire after controller restart?

--- BUILD ---

4. Implement cron expression parsing and next-fire computation
   - Use the cron crate
   - SQLite table: cron_jobs
     { cron_id, expression, job_template_id, last_fired_at,
       next_fire_at, concurrency_policy, enabled }
   - On startup AND after each leader election:
       reload all cron jobs from SQLite
       compute next_fire_at using cron::Schedule::after(Utc::now())
   - Store next_fire_at as UTC Unix timestamp (i64 milliseconds)
   - Test: '*/5 * * * *' → next_fire_at always within next 5 minutes
   - Test: '0 9 * * 1-5' → next_fire_at skips weekends

5. Implement the cron tick loop
   - tokio task that wakes at earliest next_fire_at
     (use tokio::time::sleep_until — do NOT poll every second)
   - When it wakes, for each cron job where next_fire_at <= now:
       1. Check idempotency guard:
          if last_fired_at >= current_interval_start → SKIP
          (this prevents double-fire after failover)
       2. Submit job from template
       3. Update last_fired_at
       4. Compute new next_fire_at
       5. Write both to SQLite
       6. Go back to sleep
   - Test: crash + restart scheduler at T+30s into a 1-minute interval
     → verify exactly 1 firing per minute (not 0, not 2)
   - Test: failover mid-minute → standby's cron tick does not double-fire

6. Implement job templates
   - JobTemplate { template_id, name, job_type, default_payload,
                    default_priority, default_retry_policy, tags }
   - Store in SQLite
   - gRPC RPCs: CreateTemplate, GetTemplate, UpdateTemplate,
     DeleteTemplate, ListTemplates, SubmitFromTemplate
   - SubmitFromTemplate accepts template_id + optional field overrides
   - Cron jobs reference templates
   - Test: template with default priority=5, submit with priority_override=8
     → result has priority=8

--- TEST ---

7. Double-fire prevention test — THE critical cron correctness test
   Setup: cron job firing every 1 minute, 2 schedulers (leader + standby)
   Action: at T+30 seconds into a minute → kill the leader
   Verify: exactly 1 job fires for that minute (not 0, not 2)
   Run: 5 consecutive times
   Rule: ANY double-fire = bug in idempotency guard. Fix it.
   Rule: ANY missed fire (0) after failover = bug in cron job reload. Fix it.

CRATES THIS WEEK
----------------
cron
chrono
sqlx
tokio

RISKS
-----
[CRITICAL] Storing timestamps in local timezone in SQLite —
DST transitions cause cron job to fire TWICE or NOT AT ALL
during the ambiguous hour. ALWAYS store UTC. ALWAYS.

[CRITICAL] No idempotency guard — after failover, both old leader
(if briefly recovering) and new leader may try to fire same cron job.
The idempotency guard (check last_fired_at) is the ONLY protection.

NOTES
-----
- Cron is the most-requested scheduler feature. Having a working
  cron implementation makes the project immediately relatable to
  any engineer who has used crontab, GitHub Actions schedules,
  or Kubernetes CronJobs.
