# MJDJS — Progress Context (Handoff for New Chat)

## Purpose of This Document
This summarizes everything built on the distributed job scheduler project
(codenamed MJDJS) through the end of Week 3, so a new Claude chat has full
context without re-deriving it. Provide this file alongside all current
source files (`job_data_structures.rs`, `state_machine.rs`, `worker.rs`,
`scheduler.rs`, `lib.rs`, `main.rs`) and `DECISIONS.md` when starting a new
chat.

---

## Project Overview
A general-purpose distributed job scheduler in Rust — a resume/learning
project targeting systems and infrastructure SWE roles. The scheduler
treats jobs as opaque units of work: it doesn't care what a job's payload
means, only that jobs get executed, retried on failure, and reported on
when done.

**Project name:** MJDJS · **Crate type:** library + binary · **Language:**
Rust (2021 edition)

## Roadmap Context
Four-phase, 16-week roadmap:
- **Phase 1 (Weeks 1–4):** Rust foundations + core data structures, then a
  simulated (channels-only, no networking) single-node scheduler
- **Phase 2 (Weeks 5–8):** Real gRPC/tonic server, `DashMap`-based
  concurrent state, SQLite persistence, dead-letter queue, CLI client — v1.0
- **Phase 3 (Weeks 9–13):** Distribution, etcd leader election, worker
  fault tolerance, chaos testing
- **Phase 4 (Weeks 14–16):** Observability (Prometheus/Grafana), benchmarks,
  README, demo video, launch

**Current status: Weeks 1–3 complete.** About to start Week 4 (gRPC/tonic
proto schema, workspace split — the transition from simulated to real
networking).

**Important:** several of Weeks 5–6's design documents describe the
*eventual* architecture (real workers as separate processes over gRPC,
`WorkerInfo` as pool metadata, `DashMap` instead of `Mutex<HashMap>`) —
cross-referencing that architecture against in-progress Week 3 decisions
proved valuable multiple times (see `DECISIONS.md` entries 3 and 10
especially). Worth continuing to check the fuller roadmap when a design
question comes up, not just the current week's document in isolation.

## What Was Built

### Week 1 — Complete
`Job`/`JobState`/`JobQueue` (manual `Ord` on `Job` for `BinaryHeap`
priority ordering, `created_at` tiebreaker). 19 unit tests.

### Week 2 — Complete
- **`RetryPolicy`** (`NoRetry`/`FixedDelay`/`ExponentialBackoff`) with
  jitter and `max_delay_ms` clamping, `next_delay(&self, retry_count)` as
  a pure function.
- **Generic `WorkerPool<T: Runnable>`** — deliberately chosen over
  `dyn Trait` since every worker does the same kind of thing (execute
  opaque jobs); status tracking (`PoolEntry<T>`) deliberately lives on the
  pool, not on `T`, to avoid two sources of truth.
- **`thiserror`** across all error enums (`TransitionError`,
  `QueueError`, `WorkerPoolError`).

### Week 3 — Complete
The bulk of the recent work. Highlights (full reasoning for each is in
`DECISIONS.md`):
- Fixed a busy-loop in the dispatch branch using `tokio::sync::Notify`
  (queue changes) and later `tokio::sync::watch` (worker-availability
  changes) — two distinct signals for two distinct "nothing to dispatch"
  reasons.
- `CancellationToken`-based two-phase graceful shutdown: stop accepting
  new work, keep draining `in_flight`, now bounded by a 30-second deadline
  (entry 11).
- Retry delays are now actually enforced (`waiting_retry` map + one
  spawned sleep per retry), not just computed.
- A minimal `dead_lettered` map and a `succeeded` map (both intentionally
  early/partial versions of Week 7's real DLQ).
- Two real state-machine bugs found and fixed via `let _ =` discarding
  `transition()`'s `Result` — now all transition call sites log on `Err`.
- **`Runnable::run()` is real**, replacing the old placeholder
  `simulate_job_execution` free function — `Worker` implements
  `Runnable: Clone + Send + 'static`, is cheaply cloneable with no
  mutable state of its own (matching what a future gRPC-client-backed
  worker will need to look like), and `run()` races execution against
  `cancel.cancelled()` via `tokio::select!`, returning
  `Ok(JobOutcome::{Success,Failure,Cancelled})`.
- Real timestamps (`now_millis()`, `SystemTime`-based) replaced `0`
  placeholders for `started_at`/`completed_at`.
- A 1000-job/10-worker stress test (`dispatch_order` tracking + priority
  non-increasing assertion, `succeeded`/`in_flight` completion polling
  instead of fixed sleeps).

**Full test suite: 24 tests, all passing** (`state_machine`, `worker`,
`scheduler` modules combined).

## Key Recurring Design Principles (established through repeated practice)

- **One source of truth per fact.** Caught and fixed multiple times:
  `WorkerStatus` duplicated between `Worker` and `PoolEntry`; `job_id`
  duplicated between `Worker` and `Scheduler.in_flight`; a retried job
  left in both `in_flight` and `queue` simultaneously.
- **Never hold a lock guard (`MutexGuard`, `watch::Ref`) across an
  `.await` or a call that needs the same lock.** Caused two real bugs:
  an early `assign_job`/`notified()` near-deadlock, and a genuine
  deadlock in `update_availability()` (fixed via `send_if_modified`).
- **A future with no genuine `.await` on its "nothing to do" path resolves
  instantly and busy-loops the surrounding `select!`.** The root cause of
  the very first major bug in Week 3; the general lesson that shaped
  `Notify`/`watch` usage throughout.
- **`tokio::spawn(future)` starts something running now; `.await`ing the
  resulting handle is a separate, later decision.** Collapsing both into
  one line was the root cause of two separate test-sequencing bugs.
- **Defend against unreachable states deliberately, not silently.** Every
  "this shouldn't happen" branch (`assign_job` failing, `JobExecutionError`
  with zero variants, `shutdown_cap_timer` being `None`) is logged, not
  assumed away — but genuinely unrecoverable host-level failures (a broken
  system clock) are allowed to panic loudly rather than silently corrupt
  data.

## Learning Style Note (carries forward from original README)

This is a **learn-while-building** session. The Claude assistant should:
- Give conceptual grounding before each new component, and flag any
  required reading up front.
- Provide guiding questions and hints rather than full solutions — let the
  user find bugs through their own reasoning and the compiler/test output.
- Only point out errors when reviewing *submitted* code — don't pre-solve.
- When a design decision has real weight (not just syntax), make the user
  take an explicit stance and explain their reasoning before writing code.
- Log genuinely architectural decisions (not every small naming choice) to
  `DECISIONS.md` as they happen, with the actual reasoning — not just what
  was chosen.
- The user has said motivation matters — they engage better when reasoning
  is made a little stimulating/puzzle-like rather than a dry checklist,
  especially for genuinely hard concurrency concepts.

## Immediate Next Steps (Week 4)

Per the roadmap: proto schema design, `tonic`-based gRPC server/client,
workspace split (likely separating scheduler/worker/proto into distinct
crates or a Cargo workspace). This is the transition point from the

explicitly revisiting `DECISIONS.md` entry 10 when this starts, since it
already anticipates this transition.
