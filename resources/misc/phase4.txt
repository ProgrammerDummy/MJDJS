WEEK 15: Web Dashboard UI, Profiling, Benchmark Writeup
================================================================

MILESTONE
---------
Working web dashboard with live job status.
BENCHMARKS.md with honest numbers, hardware spec, analysis
of every performance gap.

HOURS: 1.5 hrs/day weekdays · 4 hrs/day weekend
SPLIT: 40% web UI · 30% profiling · 30% writing benchmarks

--- BUILD ---

1. Build web dashboard with axum + htmx
   Pages:
   - /jobs: paginated job list (status, type, priority, created_at)
             refreshes every 2s via htmx
   - /jobs/:id: job detail — state history, retry log,
                assigned worker, execution duration
   - /workers: worker list — status, utilization, last heartbeat,
                current jobs (refreshes every 5s)
   - /dlq: dead-letter queue — list + requeue button
   - /cron: cron jobs — next fire time, last fired at, enabled toggle

   Tech choices:
   - htmx (hx-get + hx-trigger='every 2s' + hx-target + hx-swap)
     → No JavaScript framework. You need 5 htmx attributes total.
   - askama → type-safe server-side Rust templates (compile-time verified)
   - NO React, NO Vue, NO Svelte

   The /workers page is your most impressive live view —
   shows heartbeat staleness in real time.

2. Profile with cargo flamegraph — find and fix one hotspot
   Run: cargo flamegraph --bin scheduler-server under load for 30 seconds
   Identify: the widest bar (= where time is actually spent)
   Fix: the most obvious hotspot
   Re-benchmark: quantify the improvement

   Common hotspots:
   - Queue lock held too long during dispatch → minimize lock scope
   - SQLite write for every state transition → batch non-critical writes

   Include before/after flamegraph SVGs in repo under bench/

3. Run benchmarks and write honest BENCHMARKS.md

   MEASURE:
   a) Job submission throughput (RPC overhead, jobs/sec)
   b) End-to-end latency p50/p95/p99 for a trivial 1ms job
   c) Throughput scaling: 1 → 5 → 10 → 20 workers
   d) SQLite impact: with/without WAL, with/without fsync
   e) Failure detection: latency distribution over 20 runs
      (mean, p95, worst case)

   RUN ON SCHOOL HARDWARE for distributed numbers.
   Run locally for single-node numbers.

   BENCHMARKS.md STRUCTURE:
   1. Hardware spec (CPU model, RAM, disk type — SSD vs NVMe)
   2. Methodology (runs per benchmark, how cache effects avoided)
   3. Full numbers table
   4. Failure detection latency (mean, p95, p99 across 20 runs)
   5. Honest analysis of every gap

   EXAMPLE honest analysis:
   "SQLite fsync adds Xms per committed job. Without fsync it is
   Yms but data is at risk on crash. We default to fsync for
   correctness and document the tradeoff explicitly."

--- BUILD ---

1. Write the final comprehensive README

   Structure (in this exact order):
   1. One-sentence description
   2. Grafana failover screenshot (WITH annotations)
   3. Architecture diagram:
      CLI → Schedulers (2) ← etcd
      Schedulers → Workers (N)
      Arrows: job flow, heartbeat, leader election
   4. Feature checklist (checkboxes)
   5. Benchmark table
   6. Quick start (must work in 60 seconds — test on clean machine)
   7. Design decisions:
      - Why etcd (not hand-rolled consensus)
      - Why at-least-once (not exactly-once)
      - Why SQLite WAL (not Postgres)
      - Why heartbeat threshold = 3× interval
      - Why UUID v7 (not v4)
      - Why bidirectional streaming heartbeat (not unary)
   8. Chaos test results table:
      | Test | Runs | Pass Rate | Failover Time | Job Loss |
   9. Known limitations:
      - At-least-once delivery (not exactly-once)
      - Fan-out depth=1 only (no full DAG)

      - No horizontal SQLite scaling
